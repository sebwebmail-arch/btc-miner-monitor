# Capone Watcher — Business Rules & Detection Spec

## 1. Worker status classification

A worker's status is derived solely from `last_share_at` (Unix timestamp). The f2pool `status` field is **never used** — it is unreliable.

| Category | Condition | Display |
|----------|-----------|---------|
| `online` | last share < 60 min ago | ● Online |
| `offline` | last share 1h–24h ago | Offline Xh |
| `dead_recent` | last share 1–7 days ago | Dead Xd |
| `dead_mid` | last share 8–90 days ago | Dead Xd |
| `dead_old` | last share > 90 days ago | Archived |

Workers in the `No Group` category are excluded from all reports and alerts.

## 2. Daily report (07:00 Paris)

Runs via `monitor.js`. Sends one email covering both accounts.

### Offline detection
- Threshold: `last_share_at` > 60 minutes ago
- Email lists workers grouped by datacenter, per account
- Dead workers (>24h) are tracked in `history.json` but NOT re-alerted daily (they appear in the dashboard dead tracker only)

### Hashrate anomalies in report
- Reads `data/worker-issues.json` (generated by the 06:30 hashrate collector run)
- If anomalies exist, adds a second table below the offline section
- Email is sent even if there are zero offline workers but anomalies are present

### Email not sent if
- Zero offline workers AND zero hashrate anomalies

## 3. Hashrate collection (every 30 minutes)

Runs via `hashrate-collector.js`. Collects `h1_hash_rate` (1h rolling average from f2pool) for every grouped worker and appends a snapshot to `data/hashrate.json`.

### Storage
- Key format: `"cmine.031"`, `"everminer.r001"` etc.
- Each entry: `{ "ts": "ISO8601", "hr": 123456789000 }` (hr in H/s)
- Rolling window: **145 snapshots = 72h** (older entries pruned automatically)
- Workers in `No Group` are not stored

### Detection order (CRITICAL)
Detection happens **before** saving the new snapshot, so the reference is always historical data, never the current point.

1. Load existing `hashrate.json`
2. Fetch fresh workers from f2pool API
3. Run group-level drop detection
4. Run worker-level anomaly detection
5. Send alerts
6. Append new snapshots to `hashrate.json`
7. Save `alert-state.json` and `worker-issues.json`

## 4. Group hashrate drop alert

Detects sudden drops in the **total hashrate of a datacenter group** (e.g., all BitCluster workers combined).

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Reference window | Last 3 snapshots = 1h30 avg | Short enough to be reactive |
| Drop threshold | > 30% | Significant enough to indicate an incident |
| Minimum reference HR | 1 GH/s | Ignores inactive groups |
| Cooldown | 4h per group | Prevents alert spam during prolonged outage |

**Alert channels**: Telegram + email, routed per account.

**Alert key format**: `"cmine.E1"`, `"everminer.R1"` etc. (stored in `alert-state.json`)

**Alert message includes**: group name, account, drop %, before/after TH/s, timestamp in UTC + Paris time.

## 5. Worker hashrate anomaly detection

Detects individual workers that are technically "online" but performing abnormally. Runs every 30 minutes alongside group detection.

### Windows

| Window | Snapshots | Duration |
|--------|-----------|----------|
| Current | Last 6 | ~3h |
| Baseline | Snapshots 7–30 | ~3h to 15h ago |
| Minimum baseline | 12 snapshots | Required to activate (needs ~9h of data) |

### Anomaly types

**`level_drop`** — worker running at significantly reduced capacity
- Condition: `current_avg < baseline_avg × 0.60` (drop > 40%)
- AND: `current_avg > 5 TH/s` (worker is still mining, just less)
- AND: baseline_avg > 5 TH/s (worker was meaningfully active in baseline)
- Takes priority over `volatile` if both conditions are true

**`volatile`** — hashrate oscillates erratically (ON/OFF pattern, firmware issues)
- Condition A: coefficient of variation `(std_dev / mean) > 55%` on current window
- Condition B: OR `> 35% of current snapshots` are below 5 TH/s (near-zero)

### Cooldown
- 8h per worker (stored in `alert-state.json` with key prefix `"w."`)
- Alert channel: Telegram only (no email — too granular for 07:00 report)

### Output
- Saved to `data/worker-issues.json` every 30min (read by dashboard + daily email)
- Format: `{ issues: { "cmine.s21hyd007": { type, current_avg_ths, baseline_avg_ths, drop_pct, cv_pct, zero_rate_pct, ... } } }`

## 6. Alert routing per account

Each account has its own Telegram group and alert email. Both fall back to shared env vars if account-specific ones are not set.

| Account | Telegram secret | Email secret |
|---------|----------------|--------------|
| cmine | `TELEGRAM_CHAT_ID_CMINE` | `ALERT_EMAIL_CMINE` |
| everminer | `TELEGRAM_CHAT_ID_EVERMINER` | `ALERT_EMAIL_EVERMINER` |
| fallback | `TELEGRAM_CHAT_ID` | `ALERT_EMAIL` |

## 7. Dashboard data flow

```
f2pool API
    │
    ├─── monitor.js (07:00) ──→ data/history.json  ──→ index.html (issues table, trend, DC grid)
    │
    └─── hashrate-collector.js (every 30min)
              ├──→ data/hashrate.json   ──→ index.html (hashrate charts, online workers table)
              ├──→ data/alert-state.json
              └──→ data/worker-issues.json ──→ index.html (anomaly badges) + monitor.js email
```

## 8. Threshold tuning guide

All thresholds are constants at the top of `hashrate-collector.js`. After accumulating real data, adjust:

| Constant | Default | Tune if... |
|----------|---------|-----------|
| `DROP_THRESHOLD` (group) | 30% | Too many false positives → raise; missing real drops → lower |
| `LEVEL_DROP_THR` (worker) | 40% | Same logic |
| `CV_THR` | 55% | Workers with naturally variable HR trigger false alarms → raise |
| `ZERO_RATE_THR` | 35% | Adjust based on how often miners legitimately go to 0 briefly |
| `COOLDOWN_H` (group) | 4h | If incidents last longer → raise to avoid re-alerts |
| `WORKER_COOLDOWN_H` | 8h | Same |
| `REF_SNAPSHOTS` (group) | 3 (1h30) | Longer = less reactive but fewer false positives |
